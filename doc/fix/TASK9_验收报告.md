# TASK 9: 参数被重置问题 - 验收报告

## 任务概述

**问题**：用户设置的策略参数在每次选股请求后被重置为默认值

**用户参数**：
- CAP: 44（亿元）
- J_VAL: 34
- M: 19
- M1: 18
- M2: 32
- M3: 61
- M4: 118
- N: 7
- duokong_pct: 7
- short_pct: 6

## 问题分析

### 根本原因

**第一层问题**：参数单位转换导致的参数不一致
- YAML中的CAP = 44（亿元）
- Registry初始化时，通过__init__()转换为 4400000000（元）
- get_strategy()调用时，直接从YAML读取CAP = 44，覆盖了转换后的值
- 结果：参数被"重置"为YAML中的值

**第二层问题**：前端显示转换后的参数值导致的参数覆盖
- 前端从API获取参数，显示转换后的值（CAP: 4400000000）
- 用户看到这个值，认为这是正确的参数
- 用户修改其他参数并保存时，前端发送所有参数给后端
- 后端将转换后的CAP值（4400000000）保存到YAML文件
- 结果：YAML文件中的CAP被覆盖为 4400000000

### 影响范围

- `/api/strategies` - 获取策略列表
- `/api/strategy/<name>` - 获取策略详情
- `/api/select` - 执行选股（每次调用都会重新加载参数）

## 解决方案

### 修改1：改进 `strategy_registry.py` 的 `get_strategy()` 方法

重新实例化策略对象而不是直接赋值参数，确保参数经过正确的转换逻辑。

**关键改进**：
1. 重新实例化策略对象，确保参数经过__init__()的转换逻辑
2. 保留元数据和参数定义
3. 更新缓存中的策略对象

### 修改2：改进 `web_server.py` 的 `get_strategies()` 和 `get_strategy_detail()` 方法

直接从YAML文件读取原始参数值，而不是返回转换后的参数值。

**关键改进**：
1. 从YAML文件直接读取原始参数值
2. 返回原始参数值给前端（CAP: 44，而不是 4400000000）
3. 前端显示原始值，用户修改并保存时保存原始值
