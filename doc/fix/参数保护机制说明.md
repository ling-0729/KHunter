# 参数保护机制说明

## 问题描述
用户设置的策略参数（如 CAP: 44, J_VAL: 34 等）总是被重置为默认值，导致参数配置无法持久化。

## 根本原因分析

### 原因1：参数保存逻辑缺陷
**文件**：`web_server.py` 中的 `save_strategy_params()` 函数

**问题**：
- 前端 `saveConfig()` 只收集用户修改的参数
- 后端 `save_strategy_params()` 用这个不完整的参数集合覆盖整个参数配置
- 结果：未被修改的参数全部丢失，被重置为默认值

**示例**：
```
前端发送: {CAP: 44}
后端保存: {CAP: 44}  ← 其他参数（J_VAL, M, M1 等）全部丢失！
```

### 原因2：配置更新逻辑缺陷
**文件**：`web_server.py` 中的 `update_config()` 函数

**问题**：
- 直接用前端发送的配置覆盖整个 YAML 文件
- 没有保留现有的参数

## 解决方案

### 方案1：改进 `save_strategy_params()` 函数 ✓
**改进内容**：
- 只更新前端发送的参数
- 保留 YAML 中的其他参数不变
- 读取现有配置 → 更新指定参数 → 写回文件

**代码逻辑**：
```python
# 1. 读取现有配置
config = yaml.safe_load(f)

# 2. 获取该策略的现有参数
existing_params = config['strategies'][name]['params']

# 3. 只更新前端发送的参数
for param_name, param_value in params.items():
    existing_params[param_name] = param_value

# 4. 写回文件
yaml.dump(config, f)
```

### 方案2：改进 `update_config()` 函数 ✓
**改进内容**：
- 只更新指定策略的指定参数
- 保留其他策略和参数不变
- 采用与 `save_strategy_params()` 相同的逻辑

**代码逻辑**：
```python
# 1. 读取现有配置
config = yaml.safe_load(f)

# 2. 对每个策略进行更新
for strategy_name, params_update in update_data.items():
    # 3. 只更新指定的参数
    for param_name, param_value in params_update.items():
        config['strategies'][strategy_name]['params'][param_name] = param_value

# 4. 写回文件
yaml.dump(config, f)
```

### 方案3：改进 `BowlReboundStrategy.__init__()` ✓
**改进内容**：
- 添加 CAP 参数单位转换逻辑
- YAML 中的 CAP 是亿元（44），代码自动转换为元（4400000000）

**代码逻辑**：
```python
if 'CAP' in params:
    cap_value = params['CAP']
    if isinstance(cap_value, (int, float)) and cap_value < 1000:
        # 认为是亿元，转换为元
        params['CAP'] = int(cap_value * 1e8)
```

## 参数流程图

```
用户修改参数
    ↓
前端 saveConfig() 收集修改的参数
    ↓
发送 POST /api/strategy/{name}/params
    ↓
后端 save_strategy_params()
    ├─ 读取现有配置
    ├─ 只更新指定参数
    ├─ 保留其他参数
    └─ 写回 YAML 文件
    ↓
重新加载策略
    ↓
参数持久化成功 ✓
```

## 参数保护机制

### 保护级别1：参数更新隔离
- 每次只更新指定的参数
- 其他参数保持不变
- 防止参数被意外覆盖

### 保护级别2：配置文件备份
- 更新前读取现有配置
- 确保所有现有参数都被保留
- 只修改指定的参数值

### 保护级别3：类型转换验证
- 参数值转换时进行类型检查
- 转换失败时返回错误，不保存
- 确保参数类型正确
