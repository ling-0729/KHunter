# 选股执行日志改进

## 问题描述

用户在执行选股时，等待时间很长（5-10分钟），但无法看到后台的执行进度，导致不知道系统是否在正常工作。

## 解决方案

在后端增加详细的执行进度日志，让用户能通过查看日志文件了解选股的执行进度。

## 改进内容

### 1. 前端超时时间调整
- **文件**: `web/static/js/app.js`
- **改进**: 将超时时间从5分钟改为10分钟
- **原因**: 选股过程耗时较长，需要更长的超时时间

### 3. 日志输出位置

所有日志输出到 `logs/app.log` 文件，用户可以通过以下方式查看：

```bash
# 实时查看日志（Windows PowerShell）
Get-Content logs/app.log -Tail 50 -Wait

# 或者直接打开文件
notepad logs/app.log
```

## 日志级别说明

| 级别 | 说明 | 示例 |
|------|------|------|
| INFO | 重要信息 | 选股开始、加载进度、策略完成、总耗时 |
| DEBUG | 调试信息 | 单只股票分析失败 |
| WARNING | 警告信息 | 参数被修改、参数变化 |
| ERROR | 错误信息 | 加载失败、执行失败 |

## 使用示例

### 查看选股进度

1. 打开终端或PowerShell
2. 进入项目目录
3. 执行选股请求
4. 在另一个终端查看日志：

```bash
# PowerShell
Get-Content logs/app.log -Tail 100 -Wait

# 或使用 tail 命令（如果安装了 Git Bash）
tail -f logs/app.log
```

### 分析性能瓶颈

通过日志可以看到：
- 数据加载耗时
- 每个策略的执行耗时
- 每个策略的选股数量
- 总体执行时间

例如，如果某个策略耗时特别长，可以考虑优化该策略的算法。

## 性能参考

基于当前系统配置（约3600只股票）：
- 数据加载：40-50秒
- 单个策略执行：60-80秒
- 三个策略执行：180-240秒
- 总体耗时：220-290秒（3.5-5分钟）

## 后续优化方向

1. **缓存优化** - 缓存股票数据，避免重复加载
2. **并行处理** - 使用多线程或异步处理加速策略执行
3. **算法优化** - 优化指标计算，减少不必要的计算
4. **数据库优化** - 使用数据库缓存，加速数据查询
